# Namespace conventions

Plugin content is addressed as `<plugin>:<name>`. Rules the extraction follows:

1. **Skills**: invoked as `dev-pipeline:dev-pipeline`, `review-toolkit:review-lead`, `intake-toolkit:plan-interview`, etc. **User-facing slash invocations are namespaced too**: `/dev-pipeline:dev-pipeline`, `/intake-toolkit:intake`, `/dev-pipeline:pipeline-retro`, `/audit-toolkit:audit`. (Verified empirically: bare short forms like `/pipeline-retro` do NOT resolve for plugin skills once the same-named vendored copy is gone — "Unknown command". Docs and stage files must always show the qualified spelling.)
2. **Agents**: `.mjs` workflows and stage files reference agents by qualified name (`review-toolkit:security-reviewer`). Repo-local agents (config `reviewers.add`) are referenced bare — that's the disambiguation between roots.
3. **Cross-plugin dependencies are one-directional**: dev-pipeline → {review-toolkit, intake-toolkit, design-toolkit, audit-toolkit}. The four toolkits never reference dev-pipeline or each other, except intake-toolkit ← review-toolkit sharing the `reviewer-baseline` protocol (review-toolkit owns it; intake-toolkit references it qualified).

   **Workflow-script invocation (the intake/code-review exception, resolved structurally).** Per D-15, the Claude-Code-specific `.mjs` Workflow scripts are confined to the dev-pipeline plugin — `code-review.mjs` (invoked by `review-toolkit:review-lead`) and `intake-review.mjs` (invoked by `intake-toolkit:intake-orchestrator` / `decomposition-reviewer`) live there. To keep the dependency one-directional, a toolkit skill that dispatches such a workflow names it by **bare filename** (`Workflow({ scriptPath: "code-review.mjs" })`) — the Workflow tool resolves it against the installed-plugin search path — and never hard-codes a `dev-pipeline/…` path. Toolkit skills/agents likewise reference dev-pipeline artifacts (state-schema, stage definitions) by name, not by a `../dev-pipeline/…` link. This is the structural resolution of the Phase-1 rule-3 finding (intake-orchestrator/decomposition-reviewer previously hard-pathed into `.claude/skills/dev-pipeline/workflows/`).

   **Enforcement (CI).** Two greps over the four toolkits: (a) no `dev-pipeline:` namespace token anywhere; (b) no `dev-pipeline/`-path reference (`skills/dev-pipeline/`, `dev-pipeline/{workflows,stages,state-schema,doc-update}`, `../dev-pipeline/`) in toolkit **content** (`skills/**` + `agents/*.md`). Validator/eval **tools** (`scripts/check-model-tiers.sh`, the `evals/*/run*.sh` harnesses) are exempt from (b): resolving the sibling dev-pipeline plugin to scan its workflow tables is the tool's job, not a content dependency.
4. **Tolerant resolution** where runtime namespaces vary (top-level vs plugin-bundled MCP servers): probe both spellings, as `figma.mjs` already does for `mcp__figma__*` / `mcp__plugin_figma_figma__*`.
5. **Migration compatibility**: consumer repos must not carry files shadowing plugin-shipped names (the `check-reviewer-references.sh` tripwire); the old `figma-faithful:` namespace is retired in each repo's migration PR, not aliased.
6. **Restart after deleting vendored copies.** The migration deletion (removing repo-local `.claude/skills/*` / `.claude/agents/*` that shadow plugin names) must be followed by a **fresh Claude Code session** before invoking any plugin-namespaced skill. Observed on the first migrated consumer: deleting the vendored skills mid-session invalidated the session's in-memory skill registry — every `Skill(<plugin>:<name>)` invocation returned "Unknown skill" for the rest of that session, while the installed plugins remained fully valid on disk (`claude plugin details <plugin>@second-shift` lists the skills; a fresh session resolves them). There is no in-session reload. Sequence a migration as: enable plugins → delete vendored copies → commit → **restart the session** → run the validation gates (selftests, dry-run ticket) in the new session.
